### 1. Сценарии использования
Для скаут-агента основные сценарии:
- **Извлечение тактических определений:** Агент, получив запрос на анализ игрока в конкретной роли, должен найти в базе знаний именно те критерии, которые стоит применить в анализе именной этой роли.
- **Уточнение методологии:** Нахождение правил интерпретации статистических данных, таких как xG. Например «критические показатели xG» для нападающего.
- **Рассмотрение принципов скаутинга:** Сопоставление анализа с типичными принципами выделения конкретных игроков.

### 2. Метрики и их обоснование
Для анализа ролей и методологии нам важно, чтобы агент получил весь набор нужных критериев из файла, а не «лучший» абзац на первом месте. Поэтому выбираем следующие метрики:
- **Precision@k (Точность):** Доля релеватных документов среди k найденных. Важна, чтобы агент не мусорил в контекст лишней информацией.
- **Recall@k (Полнота):** Найдены ли все необходимые части инструкций из файла. Если найдена только половина или меньше, анализ будет неполным.

### 3. Формирование данных (Qrels)
Для объективной оценки ретривера будет использован подход Qrels. Процесс формирования данных включает три этапа:
1. **Подготовка корпуса:** Текст из scout_knowledge.txt разбивается на логические чанки с помощью RecursiveCharacterTextSplitter. Каждому чанку присваивается уникальный идентификатор.
2. **Генерация тестовых запросов:** Создается набор из 20–30 контрольных вопросов, охватывающих все темы из базы знаний (роли, метрики, принципы).
3. **Создание «Золотого стандарта»:** Вручную размечается таблица соответствия — какой вопрос должен находить какой чанк.

### 4. Анализ результатов
**Результаты тестирования:**
- Mean Recall@3: 0.56 (Полнота)
- Mean Precision@3: 0.19 (Точность)

**Интерпретация метрик:**
1. **Полнота:** Система успешно находит релевантный контекст в 56% случаев. Случаи, когда Recall = 0, показывают, что легкая модель эмбеддингов чувствительна в формулировкам. Если в запросе используются описательные обороты вместо точных терминов из базы, семантическая близость векторов оказывается недостаточной для попадания в топ-3.
2. **Точность:** Низкий показатель точности является математически ожидаемым при конфигурации k=3. Тк для большинства запросов в тестовом наборе предусмотрен только один эталонный документ, максимально возможная точность ограничена значением 1/3. Значение 0.19 указывает на то, что в части успешных сессий релевантный документ находится, но сопровождается шумом из соседних чанков.

**Улучшить можно несколько вещей:**
- Оптимизировать базу знаний, т е добавить синонимы, расшифровки и тд.
- Внедрить гибридный поиск, векторный и по ключевым словам к примеру, позволить находить специфические термины.
- Добавление второго этапа фильтрации для Топ-10 результатов поиска. Это позволит более мощной модели детально сопоставить запрос пользователя с контекстом чанков, что значительно повысит метрику Recall и отсеет нерелевантные срабатывания. То есть Reranker.
