# Артефакт: Подходы для оценки критериев

### 1. Фактическая согласованность
* **Класс метода:** Критерий проверяется без двусмысленности -> **Точная оценка**.
* **Обоснование:** Для оценки этого критерия мы имеем эталонные данные в API, к которым система обращается. Таким образом мы будем использовать измерение схожести. Конкретно для сухих цифр и коротких ответов используем точное совпадение, а для имен, названий и прочего используем лексическое сходство.

### 2. Точность маршрутизации
* **Класс метода:** Критерий проверяется без двусмысленности -> **Точная оценка**.
* **Обоснование:** Для оценки точности маршрутизации мы проверяем напрямую функциональную корректность, так как нам нужно четко понимать – правильно ли агент выбрал инструмент для выполнения запроса.

### 3. Безопасность
* **Класс метода:** Критерий проверяется с двусмысленностью -> **Субъективная оценка**.
* **Обоснование:** Для оценки безопасности нам нужно учитывать как и безопасность данных пользователя, так и оценивать насколько ответ является не токсичным и не задевает какие-либо этические вопросы. В этом случае мы можем использовать комбинированный подход: AI-as-a-Judge использовать для быстрой проверки по правилам безопасности, а человеческую оценку для финальной верификации и соответствию человеческой морали.

### 4. Точность извлечения сущностей
* **Класс метода:** Критерий проверяется без двусмысленности -> **Точная оценка**.
* **Обоснование:** Для оценки точности извлечения сущностей есть надежные эталонные данные – четко прописанные в системе версии написания имен и названий -> используем измерение схожести, а именно в основном лексическое сходство, для проверки правильности написания имен, без опечаток, а точное совпадение для ID игрока, т.к. нужно соблюдать точность цифр в нем, иначе ответ будет неверным.

### 5. Полнота и релевантность ответа
* **Класс метода:** Критерий проверяется с двусмысленностью -> **Субъективная оценка**.
* **Обоснование:** Для оценки полноты и релевантности ответа мы можем использовать AI-as-a-Judge, чтобы проверить логическое и фактическое сходство запроса пользователя и итогового анализа системы. Для этого можно составить «чек-лист» для AI, которая будет проверять «Пользователь запросил проверку этого, этого и этого пункта – есть ли в итоговом анализе и вердикте эти пункты?».

### 6. Логическая обоснованность
* **Класс метода:** Критерий проверяется с двусмысленностью -> **Субъективная оценка**.
* **Обоснование:** Для оценки логической обоснованности мы также можем использовать AI-as-a-Judge, дав ему нужные инструкции по проверке логики ответа. AI будет обращаться к исходным данным и оценивать насколько итоговая аналитика логичная, относительно этих данных.

### 7. Глубина инсайтов
* **Класс метода:** Критерий проверяется с двусмысленностью -> **Субъективная оценка**.
* **Обоснование:** Для оценки глубины инсайтов можно использовать человеческую оценку, т.к. в этом случае нужно увидеть насколько система видит скрытые тренды, а не просто в сухую анализирует статистику.

### 8. Стабильность формата
* **Класс метода:** Критерий проверяется без двусмысленности -> **Точная оценка**.
* **Обоснование:** Для оценки стабильности формата все просто, мы используем функциональную корректность, чтобы точно проверить совпадение структуры итогового отчета с заданной структурой JSON.
